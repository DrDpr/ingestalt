Fixed the canvas superimposition bug where new canvases would overwrite nodes in "All Nodes" view.

## Root Cause
When ingesting into a new canvas, the system was:
1. Creating nodes with temporary `workspaceId = folderName`
2. Finding ALL nodes with that workspace ID (including previously ingested ones)
3. Updating them all to the new canvas's workspace
4. This caused existing nodes to be "stolen" from their original canvases

## Solution

### Modified Return Types
**ingest-client.ts & ingest-fsa.ts**
- Changed return type from `number` to `{ count: number; nodeIds: string[] }`
- Now tracks which specific nodes were just ingested
- Returns the IDs of newly created nodes

### Precise Node Updates
**Toolbar.tsx**
- Uses the returned `nodeIds` array to update ONLY newly ingested nodes
- No longer searches for nodes by workspace name (which could match old nodes)
- Updates edges only if they connect to the newly ingested nodes
- Prevents "stealing" nodes from other canvases

## How It Works Now

**Scenario: Multiple ingestions of same folder**
1. Ingest "project1" into Canvas A → Nodes get unique IDs, assigned to workspace A
2. Create Canvas B, ingest "project1" again → NEW nodes created with different IDs, assigned to workspace B
3. Canvas A still shows its original nodes
4. Canvas B shows its own copy of the nodes
5. "All Nodes" shows both sets without conflict

**Scenario: Switching between canvases**
1. Canvas A has nodes from "project1" (workspace A)
2. Canvas B has nodes from "project1" (workspace B)
3. Switch to Canvas A → Shows only workspace A nodes
4. Switch to Canvas B → Shows only workspace B nodes
5. Switch to "All Nodes" → Shows both workspaces' nodes
6. No superimposition or overwriting

The fix ensures complete isolation between canvases with precise node tracking during ingestion.

## Root Cause
The parser was reusing node IDs from the markdown frontmatter. When the same files were ingested into different canvases, they would have identical IDs, causing them to overwrite each other in the database instead of creating separate instances.

## Solution

### 1. **parser.ts** - Added forceNewId parameter
- New optional parameter `forceNewId` (default: false)
- When true, always generates a fresh unique ID
- Ignores any existing `id` field in the markdown frontmatter
- Enables true data forking

### 2. **ingest-client.ts** - Propagate forceNewIds flag
- Added `forceNewIds` parameter to `layoutAndPersist` (default: true)
- Added `forceNewIds` parameter to `ingestWorkspace` (default: true)
- Passes the flag through to the parser
- Ensures all ingestions create new instances by default

## How Data Forking Works Now

**Scenario: Fork the same project into multiple canvases**
1. Canvas A: Ingest "project1" → Creates nodes with IDs like `node_1234_abc`
2. Canvas B: Ingest "project1" again → Creates NEW nodes with IDs like `node_5678_def`
3. Both canvases have independent copies of the same data
4. Editing nodes in Canvas A doesn't affect Canvas B
5. Each canvas can evolve its data independently

**Scenario: Multiple sources in one canvas**
1. Canvas A: Ingest "project1" → Nodes created
2. Canvas A: Ingest "project2" → More nodes added
3. All nodes coexist in Canvas A with unique IDs
4. No conflicts or overwrites

**Scenario: Re-ingesting into same canvas**
1. Canvas A: Ingest "project1" → First set of nodes
2. Canvas A: Ingest "project1" again → Second set of nodes (duplicates)
3. Both sets exist independently in Canvas A
4. Allows versioning/comparison workflows

The system now supports true data forking, enabling experimentation, versioning, and independent evolution of data across different canvases.
